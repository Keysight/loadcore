About This Release

With LoadCore 3.2 release we are introducing a whole new level of application traffic richness, by enabling hundreds of predefined application flows.
LoadCore VoNR offering is further expanded through 3GPP IPsec support, Video calls, SIP over UDP transport and many more.
Control plane wise, LoadCore 3.2 brings in support for more SBA nodes (SCP, NEF), 4G nodes (SGW), TSN on control plane and many other new procedures and enhancements to existing procedures.
Further cloud evolution is enabled through AWS EKS support, embedded licesning and application traffic support for containers.

  • Fixed an issue causing spikes in Application traffic general dashboard

Installation Notes

LoadCore 3.2 provides a new middleware image and an upgrade file. Please also note that upgrading a setup from pre-LC3.0 version to LC3.2 is not supported.
LoadCore 3.2 also comes with new agent images, no upgrades here.
AWS and Azure images are not distributed on the product download page. AWS image can be found on the AWS Marketplace, and both (AWS and Azure) can be shared by request by the engineering team. Please contact engineering at email LoadCore-Support to request access to these images.

The REST API dashboards names have been changed. For automation users, in case your harness is checking the statistics, please use the REST API stats migration script to automatically convert the old naming into the new one.
Please follow these steps to create a setup for running the product:

If a LC3.0 or 3.2 setup is not yet deployed, please use the below steps to deploy LC3.2 and configure an account

  1. Deploy the UI/middleware cluster OVA file
  2. Deploy the licensing OVA file
  3. After deployment you can access the UI using the MW machine IP and access it using a browser: https://XX.XX.XX.XX. If shortly after deploying the MW the UI is not accessible, you need to wait a couple of minutes for all the pods to start up.
  4. To access the UI, you will be prompted for a username and password. You will need to register a new user and then use it each time you log in.
  5. Connect to the LC3.2 with your account. If a LC3.2 setup is not already available please use the above steps to deploy a new setup

If a LC3.0 setup is deployed, please use the below steps to upgrade:

    Apply the patch using the UI:

  1. After connecting to the LC3.0 setup, find the "Gear menu" in the upper right corner and click on it to bring up the menu
  2. Then, click on "Administration" > "Software Updates"
  3. Click the "Browse" button and use the opened control to find the LC3.0 upgrade tar on the disk. If the LC environment is cloud (not on premise) please use the cloud tar (LoadCore-MDW-cloud-BUILD_VERSION.tar).
  4. Click on the "Start Update" button to apply the patch and wait for the procedure to end.
  5. Refresh your LoadCore page in the broswser
  6. Access the same menu and check the version of the setup again to make sure that the patch has been applied successfully.

    Apply the patch using the KCOS CLI:

  1. Connect to the LC3.0 setup with either SSH (on port 22: admin/admin user) or directly through the console (console -> admin/admin user). This will make available the KCOS CLI.
  2. Copy the LC3.2 update file to the MW using SCP. (admin/admin user). If the LC environment is cloud (not on premise) please use the cloud tar (LoadCore-MDW-cloud-BUILD_VERSION.tar).
  3. Give the following command in the KCOS CLI pointing to the update tar that was copied on the MW machine: kcos deployment offline-install ./LoadCore-MDW-BUILD_VERSION.tar
  4. Wait for the process to finish. This might take several minutes.
  5. Refresh your LoadCore page in the broswser
  6. Access the UI and check the version of the setup to make sure that the patch has been applied successfully.

For deploying the agents, please follow these steps:

  1. Deploy the agent OVA file
  2. Add another NIC to the agent VM from the hypervisor configuration window. This will be the test NIC.
  3. Log in to the agent VM and configure the management interface, if needed
  4. Run the agent-setup.sh script and follow the steps in the interactive setup. This step is required to connect the agent to the middleware/UI cluster.
    5. #  sudo ./agent-setup.sh
  5. At this point, you should be able to see the newly created agent in the UI
Repeat these steps to deploy as many agents as required.

Known Limitations

    Application Infrastructure

  • IRat topology is no longer supported and has been removed from LoadCore UI
  • LoadCore configuration files generated in versions older than 2.3.0 are no longer supported. In order to use such config files, the workaround is to import and export them into a LoadCore 2.3.0 or 3.0.0 instance.
  • When checking license statistics, in the application column appears LoadCore N/A instead of LoadCore
  • When taking licenses from SLUM, instead of username, it appears an hexa code
  • Predefined flows are not working when the agent is connected to the MW without using the LoadBalancer public ip
  • Sometimes MW upgrade from 2.1.1 fails because of NG-RAN Simulation session that does not pass the upgrade. Workaround is to delete the NG-RAN Simulation session(s) before the upgrade.
  • On SRIOV setups using Intel NIC we can use a maximum of 4 cores
  • On SRIOV setups using Intel NIC, the recommendation is to use a MTU equal to 1480 when configuring VLANs on the N3 interface
  • Deploying a new middleware, in rare occasions, fails because of the weavenet pod gives an error. The workaround is to delete the weavenet pod which will trigger the pod to be recreated correctly
  • Sometimes a test could fail to start with 'Lizard communication error' if the required agents were rebooted from UI before starting the test. As a workaround the user should wait for about a minute then start the test again.
  • Sometimes the LC MW becomes unavailable after a reboot. Workaround: reboot the LC MW again.
  • Sometimes the LC MW UI hangs while it gets updated. Workaround: wait several minutes, refresh the UI and check the Keysight Load Core version.
  • The Per-UE Statistics feature only works with Stateless UDP flow. The Per-UE Statistics will show up as empty in UI if running an Application Traffic test.
  • The Per-UE Statistics feature is currently only supported for the NG-RAN application and will not generate CSVs for any other nodes when running a test with this feature.
  • The Process of updating LoadCore MDW from Web UI in a cloud environment might not reach the end. The workaround is to refresh the Web UI and check the new versions.

    • Control Plane

  • Advanced stats for control plane latency don't increment when per UE stats are enabled
  • No route is added for the SGW standalone if any of the endpoints are in different networks
  • When N14 is enabled, only 2 AMF ranges can be configured
  • No support for scenarios where the source AMF changes the active KAMF (horizontal KAMF derivation)
  • The N2 Handover Execution Phase messages might intercalate with the Mobility Registration Update messages resulting in a different order than the one defined in TS23502 4.9.1.3.3
  • AMF Distribution on multiple agents is not supported
  • N2 handover tests with AMF change and active N14 interface in which one UE performs more than one handover per second might result in handover failure
  • Only one type of handover (N2HO with AMF change) is supported while N14 is enabled. N2 Handover with N14 and 5GS to EPS / EPS to 5GS handover is supported with the condition that only one AMF is involved in IRAT handover and both AMFs have N26 enabled.
  • Enter/Exit Idle or Paging are not supported while running with N14 interface enabled
  • The DUT AMF must have the correct GUAMI value configured (it should match the one configured on the actual AMF DUT). Otherwise, the N14 connection will not be established.
  • For legacy configurations the NSSAI and TAI will be be empty lists. In N14 tests, NSSAI and TAI configuration is mandatory. In order to have successful UE registrations, please make sure the NSSAIs configured on the UE and AMF match.
  • N14 interface is mandatory when running AMF nodes on different agents during N2 handover
  • Increasing the number of HTTP connections on AMF and SMF does not improve performance
  • No support for notifications from NEF in this release
  • On NEF for AF Session with QoS service the UE identification is done only based on UE IP
  • On NEF for Service Parameter service the UE identification is done only based on UE IP or GPSI
  • On NEF for Device Triggering service the UE identification is done only based on MSISDN
  • On NEF the following API calls are not implemented: /3gpp-as-session-with-qos/v1/{scsAsId}/subscriptions, /3gpp-service-parameter/v1/{afId}/subscriptions, /3gpp-device-triggering/v1/{scsAsId}/transactions
  • On SCP Delegated Discovery (Model D) is not supported
  • No support for Routing Binding Indication on SCP
  • For next hop SCP, if SCP receives from NRF a port, it will use it, otherwise it will default to 80 (HTTP) or 443 (HTTPS) depending on the protocol used by SCP
  • For UPF Isolation topology when mixing Subscriber ranges with IPv4 or IPv6 with dual-stack ranges (IPv4v6) ensure the dual stack ranges are defined last
  • UPF Isolation topology: UPF passthrough is not supported in tests with dual stack IP
  • On N26 interface the GTPv2 messages always contain the recovery IE
  • For idle mobility tests, in case of a race condition between the mobility event and the exit idle event, only the mobility event will be executed
  • For gNodeb+eNodeB simulation, create/delete QoS flows objective is skipped when the UE is attached in 4G
  • UL/DL data is sent only to one UPF while running with SSC mode 2 and SSC mode 3
  • Execution with 2 or more DNNs is not recommended while running with SSC mode 2 and SSC mode 3
  • Port instability could be reported while running with high number of UEs on SSC mode 3
  • “PDU session release UE originating failed” could be incremented while running with high number of UEs on SSC mode 2
  • Secondary objectives may not get a chance to start when the primary objective is set to "subscribers per second" and the hold time value is set to one second. Set a longer hold time to allow the secondary objectives to start
  • Test will not start if the connection start rate can't be reached and maintained
  • PCF initiated create/delete QoS flows objective should not be run together with other secondary objectives
  • NSSF response to GET Network Slice Selection request doesn't contain configured NSSAI when the requested NSSAI includes an S-NSSAI that is not valid in the serving PLMN.
  • Multiple registration messages for UDM are sent to NRF while the topology contains two UE subscriber ranges
  • For N2 Handover with AMF change, the S-AMF does not send the update SM context request to SMF in case of failure on T-AMF
  • For N2 Handover with AMF change testing, it is recommended to have only the handover as a secondary objective
  • All SBA nodes running on the same agent use the same HTTP/HTTPS server ports (80/443)
  • Unexpected behavior can be encountered when using duplicate UE IP addresses
  • UPF does not respond to session deletion retries
  • Increment for RAND parameter does not work as expected when the value overflows
  • Use remote F-TEID allocation option needs to be set when running the SMF node from UPF isolation topology on multiple agents
  • PFCP association release timeouts may be present in full core topology scenarios where simulated SMF and UPF are running on separate agents
  • SMF simulation from full core topology responds with error cause 404 to UDM initiated notifications
  • Statistics for default QoS flow activation procedures represent initial session establishment procedures and one procedure might activate more than one QoS flow.
  • UPF isolation: N4-SMF SLAAC message stats are not populated when their value is 0
  • UPF isolation: No support for unsolicited RAs
  • UPF isolation: SLAAC is not supported in N9 or uplink classifier tests
  • UPF isolation: SLAAC is not supported on scenarios with HTTPS flows
  • UPF isolation: When running consecutive tests without and then with "use remote F-TEID allocation", the test will no longer start because of error "Test Stopped: Agents for client not configured within 1m0s". At a new run the test will then start.
  • Session report for 5G scenario works only for one QoS flow
  • Only one MME can be simulated in full core topology
  • gNodeB simulation: For network initiated QoS flow modification, none of the parameters changed by the network initiated QoS flow modification will be enforced
  • gNodeB simulation: For network initiated QoS flow modification, the NG-RAN node supports handling the QoS flow modification procedure only for one PDU session per procedure
  • gNodeB simulation: For network initiated QoS flow modification, the NG-RAN node supports handling the QoS flow modification procedure containing only one type of action per procedure (create QoS flow/modify existing QoS flow/release QoS flow)
  • gNodeB simulation: For network initiated QoS flow modification, for existing QoS flows in order to ensure predictability, the interval between the creation and deletion of the QoS flow should be large enough to support the successful finalization for the modification of the existing QoS flow
  • In SBA Tester, when having SMS and PCF primary objectives at the same time, the test will have an undefined behavior.
  • In SBA Tester, when having SMS objective, only one IP address should be configured on the test interface.
  • The SMS activation procedure in SBA Tester supports only the 3gpp-access access type.
  • SBA Tester is not doing any action when it receives SMS specific errors. SBA Tester will only verify the validity of the SMS message.
  • SBA Tester will send error message when TI doesn’t match and will not wait for a message with correct TI.
  • SBA Tester doesn’t support STATUS-REPORT processing. Receiving this message will result in an error response 400 and cause “SMS_PAYLOAD_ERROR”.
  • SBA Tester doesn’t support two or more SMS procedures in the same time in a UE range (mobile originated or mobile terminated).
  • SMSF node will originate SMS regardless of how many mobile originated SMS received.
  • In SBA Tester, when using ConvergedChargingSMF2CHF primary objective, only the first SMF simulated node will generate requests even if more SMF nodes are configured.
  • NG-RAN doesn’t support two or more SMS procedures at the same time on the same UE range (mobile originated or mobile terminated).
  • NG-RAN doesn’t support STATUS-REPORT processing. Receiving this message will automatically trigger mobile terminated procedure or continue mobile originated procedure.
  • The only error process by NG-RAN is CP-ERROR and receiving this error results in the stop of the SMS procedure.
  • On NG-RAN, when received TI that doesn’t match the expected one, the procedure will continue without reporting this error.
  • AMF doesn't send PDU session establishment reject to UE if AMF failed to discover SMF.
  • AMF node sends wrong AMF UE ID in UE context release in case of AMF failed to discover UDM.
  • Only one query with target-nf-type set to SMF is executed by AMF (for first PDU session) while running a test with two or more DNNs. There is only one SMF node supported per agent.
  • The minimum number of queries (per type of discovery) is equal to the minimum between UE count and CPU core count when requester-nf-type is PCF or AUSF, or minimum between UE count, HTTP connection count and CPU core count when requester-nf-type is AMF or SMF.
  • The AMF node does not stop the current procedure (PDU session establishment or PDU session modification) in case SMF fails to discover AMF.
  • IPv6 SLAAC is supported only on RAN node in full core topology. SMF node in full core topology does NOT support IPv6 SLAAC
  • UDM does not support authentication results for multiple registrations
  • AUSF does not signal authentication result removal to UDM
  • If UEs are registered for emergency services and periodic registration update is enabled, test might become unresponsive.

    • User Plane

    Infrastructure & Data

  • Application traffic needs different agents for initiator and terminator sides
  • Running the whole topology with application traffic on a single agent is not supported
  • Running a full core test with passthrough activated and VLAN set on the N6 UPF interface cause the connection between RAN and AMF to be lost
  • UPF is not receiving traffic in case the RAN N3 interface is the 2nd test interface - UPF passthrough topology on container agent
  • UPF passthrough is sending downlink traffic from DN on intervals
  • Only one IP can be configured as gateway on NG-RAN and UPF passthrough settings
  • Lost user plane packets may be encountered when running enter/exit idle or paging secondary objectives together with UPF passthrough
  • DN UDP traffic ports are not configured unless at least a stateless UDP flow is configured with desired traffic ports
  • All packets coming on the passthrough interface will be mapped on the default flow on N3 interface
  • TCP resets are seen for the first iteration on single UE tests when running multiple times without service restart
  • No QoS flow enforcement of rates (AMBR/MFBR) for traffic objective.
  • FTP flows can only be used for upload and the data port for FTP flows is randomly chosen at run-time and cannot be configured
  • Using large HTTP page sizes (e.g. 1GB) is not recommended
  • When running single HTTP/HTTPS flow with persistent TCP connection option enabled, a lower number of UEs will be active for userplane traffic during the test duration as compared to the configured number of UEs. When running with Persistent TCP connection option disabled, all configured UEs will be active for userplane traffic.

    • Media

  • In tests with Voice application traffic, only one agent is supported per node
  • Voice application traffic is working only with the active subscribers primary objective
  • Only create/delete QoS flows and handovers secondary objectives are working with Voice application traffic. Other objectives like idle and paging are not working.
  • For tests with Voice application traffic, EVS full header is working with only one audio frame per RTP packet
  • On EPS fallback, the first few RTP packets are sent on the default 5G QoS flow until the 4G bearer is created.
  • H.264 is the only video codec supported for VoNR
